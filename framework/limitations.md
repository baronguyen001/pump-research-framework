# Limitations

This file is the most important part of the repo.

## Survivorship bias

The visible winners are easy to list. The dead tokens are harder to count. Any
framework built from SHIB, PEPE, WIF, BONK, SOL, HYPE, or VIRTUAL is biased
toward stories that survived long enough to be studied.

The case studies are selected examples, not a base-rate sample. Treat every
score as a hypothesis, not evidence of positive expectancy.

## Hindsight bias

"Whisper phase" sounds obvious after the fact. In real time, it often looks like
noise, spam, or a small group talking to itself. Stage classification is lagging
and subjective.

## Tool history

Modern social and on-chain tools became much better after 2023. A framework that
uses today's dashboards cannot be cleanly backtested across 2019 or 2020 without
changing what information was available at the time.

## The 5/30 problem

The 5x-in-30-days anti-FOMO rule blocks obvious late chases. It also blocks some
of the biggest runners. YFI, SHIB, PEPE, and WIF-style moves can run faster than
any cautious framework.

This is a tradeoff, not a bug. The rule is for survival, not for catching every
outlier.

## Memecoins do not fit cleanly

Memecoin rallies can be driven by image, timing, social reflexivity, and exchange
access before on-chain quality looks good. The on-chain layer helps avoid bad
contracts and concentration, but it will miss many viral moves.

## No top-of-cycle oracle

The framework has take-profit ladders and sell-the-news warnings. It does not
know when a full cycle is ending. Most paper gains disappear because the exit was
not written before the crowd arrived.

## Automation risk

Automating weak assumptions makes them faster, not better. This is why the repo
ships as a thinking framework plus a by-hand scorer. If you automate it with
Trawlkit, validate the source data, lag, false positives, and kill criteria with
your own logs.
